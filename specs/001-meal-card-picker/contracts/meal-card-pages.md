# UI 契約：餐點卡牌頁面與表單

本文件描述 Razor Pages 對使用者公開的頁面、查詢參數、表單欄位與回應行為。這些契約用於實作、測試與驗收，不代表 REST API 設計。

---

## 1. 首頁抽卡頁

### Route

- `GET /`
- `POST /?handler=Draw`

### GET `/`

#### 用途

顯示餐別選擇區、抽卡按鈕、最近一次抽卡結果區塊與導向卡牌庫的入口。

#### 頁面內容契約

| 區塊 | 內容 |
|------|------|
| 餐別選擇 | 早餐 / 午餐 / 晚餐單選或下拉選單 |
| 操作按鈕 | 「抽一張」 |
| 結果區 | 成功時顯示卡牌名稱、餐別、完整描述；未抽卡時顯示初始提示 |
| 訊息區 | 顯示驗證失敗或空卡池訊息 |

### POST `/?handler=Draw`

#### 表單欄位

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `SelectedMealType` | `MealType` | 是 | 要抽取的餐別 |
| `__RequestVerificationToken` | `string` | 是 | Razor Pages Anti-Forgery Token |

#### 成功回應

- HTTP 200（回傳同頁）
- 顯示一張屬於所選餐別的卡牌
- 結果區需包含：
  - 卡牌名稱
  - 餐別
  - 完整描述

#### 失敗 / 空狀態

| 情境 | 預期行為 |
|------|----------|
| 未選擇餐別 | 顯示「請先選擇餐別。」並保留原頁 |
| 餐別沒有卡牌 | 顯示「目前沒有可抽取的餐點。」且不得顯示假結果 |
| 表單驗證失敗 | 顯示對應欄位或摘要錯誤訊息 |

## 2. 卡牌列表與搜尋頁

### Route

- `GET /Cards`

### Query Parameters

| 參數 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `keyword` | `string` | 否 | 餐點名稱關鍵字，大小寫不敏感部分比對 |
| `mealType` | `MealType` | 否 | 餐別篩選 |

### 頁面內容契約

| 區塊 | 內容 |
|------|------|
| 搜尋表單 | 關鍵字輸入框、餐別選擇、搜尋按鈕、清除條件按鈕 |
| 列表摘要 | 每張卡牌預設只顯示名稱與餐別 |
| 操作欄 | 查看詳情 / 編輯 / 刪除 |
| 空狀態 | 查無結果時顯示明確提示 |

### 搜尋結果規則

- 僅指定 `keyword`：依名稱做大小寫不敏感部分比對。
- 僅指定 `mealType`：只顯示該餐別卡牌。
- 同時指定：僅顯示同時符合兩者的卡牌。
- 無條件：顯示全部卡牌摘要。

### 空結果訊息

- 顯示「查無符合條件的餐點卡牌。」

## 3. 新增卡牌頁

### Route

- `GET /Cards/Create`
- `POST /Cards/Create`

### POST 表單欄位

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `Input.Name` | `string` | 是 | 餐點名稱 |
| `Input.MealType` | `MealType` | 是 | 餐別 |
| `Input.Description` | `string` | 是 | 描述 |
| `__RequestVerificationToken` | `string` | 是 | Anti-Forgery Token |

### 成功行為

- 儲存成功後導回 `/Cards`
- 新卡牌可立即被瀏覽、搜尋與抽卡使用

### 驗證與錯誤訊息

| 情境 | 預期行為 |
|------|----------|
| 任一必填欄位缺漏 | 在欄位旁與摘要區顯示 zh-TW 錯誤 |
| 與既有卡牌重複 | 顯示「相同內容的卡牌已存在，請勿重複新增。」 |
| 寫入失敗 | 顯示通用錯誤訊息，保留使用者輸入 |

## 4. 編輯卡牌頁

### Route

- `GET /Cards/Edit?id={cardId}`
- `POST /Cards/Edit?id={cardId}`

### GET 契約

- `cardId` 必須存在且對應到現有卡牌
- 畫面預載既有 `Name`、`MealType`、`Description`
- `Id` 不顯示為可編輯欄位

### POST 表單欄位

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `id` | `string` | 是 | 卡牌 ID |
| `Input.Name` | `string` | 是 | 更新後名稱 |
| `Input.MealType` | `MealType` | 是 | 更新後餐別 |
| `Input.Description` | `string` | 是 | 更新後描述 |
| `__RequestVerificationToken` | `string` | 是 | Anti-Forgery Token |

### 成功行為

- 儲存成功後導回 `/Cards`
- 後續搜尋與抽卡應立即反映更新後資料

### 錯誤契約

| 情境 | 預期行為 |
|------|----------|
| `id` 不存在 | 顯示 Not Found 或導回列表並提示資料不存在 |
| 送出後形成重複卡牌 | 顯示重複錯誤，保留編輯內容 |
| 必填欄位無效 | 顯示欄位驗證訊息 |

## 5. 刪除確認頁

### Route

- `GET /Cards/Delete?id={cardId}`
- `POST /Cards/Delete?id={cardId}`

### GET 契約

- 顯示即將刪除卡牌的名稱、餐別與描述摘要
- 明確要求使用者再次確認

### POST 表單欄位

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `id` | `string` | 是 | 卡牌 ID |
| `confirmDelete` | `bool` 或明確提交按鈕 | 是 | 使用者確認刪除意圖 |
| `__RequestVerificationToken` | `string` | 是 | Anti-Forgery Token |

### 成功行為

- 刪除成功後導回 `/Cards`
- 該卡牌不得再出現在列表、搜尋與抽卡結果中

### 錯誤契約

| 情境 | 預期行為 |
|------|----------|
| 使用者未確認刪除 | 不執行刪除，回到確認頁 |
| `id` 不存在 | 顯示資料不存在提示 |
| 寫入失敗 | 顯示通用錯誤訊息 |

## 6. 共通 UI / 安全契約

### 驗證與訊息

- 所有使用者可見訊息均以繁體中文顯示。
- 所有新增 / 編輯 / 刪除表單皆同時具備：
  - Server-side validation
  - jQuery Unobtrusive Validation
  - 驗證摘要或欄位旁提示

### 安全要求

- 所有 POST 表單都必須包含 Anti-Forgery Token。
- 顯示使用者輸入時依賴 Razor 預設 HTML encoding，禁止輸出未編碼內容。
- 正式環境使用 HTTPS 與 HSTS。

### 導覽要求

- `_Layout.cshtml` 至少提供：
  - 首頁抽卡入口
  - 卡牌列表入口
  - 新增卡牌入口
